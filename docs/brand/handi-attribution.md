# Handi Attribution and Confirmed Tracking

## Purpose

This layer keeps Campaign OS identifiers, UTMs, and confirmed backend milestones aligned without introducing a new CDP or an opaque attribution engine.

The goal is pragmatic:

- browser analytics stays fast and lightweight
- backend-confirmed conversions become more reliable
- Campaign OS identifiers survive redirects and multi-step journeys

## Persistence Strategy

The browser now persists three attribution layers:

- `first_touch`
- `last_touch`
- `campaign_context`

Storage currently used:

- `localStorage` for client-side continuity
- cookies for server-readable continuity

This makes it possible for backend-confirmed routes to recover Campaign OS context even if the final conversion happens after auth redirects or multi-step flows.

## URL and CTA Propagation

Do not append Campaign OS IDs and UTM fields manually in components.

Use:

- `lib/analytics/url-context.ts` for low-level URL builders and request readers
- `lib/analytics/cta-builders.ts` for shared CTA/auth URL builders
- `lib/analytics/campaign-linking.ts` for client-side CTA helpers based on current attribution state

## Browser vs Server Event Policy

Browser-only events cover:

- views
- CTA clicks
- landing interactions
- funnel starts
- intermediate client-side actions

Server-confirmed events cover:

- records successfully created in backend
- payment confirmations validated by webhook
- admin exports/downloads generated by server routes

Current confirmed event family:

- `request_created_confirmed`
- `pro_apply_completed_confirmed`
- `fee_paid_confirmed`
- `export_package_downloaded_confirmed`
- `download_bundle_downloaded_confirmed`

## Dedupe Policy

This phase does not attempt advanced event stitching.

Current rule set:

- browser events carry `event_source=browser`
- confirmed backend events carry `event_source=server`
- confirmed events use distinct names instead of reusing browser names
- `event_id` and `correlation_id` are attached when practical

This avoids the most obvious double-counting cases without pretending that cross-device or cross-channel dedupe is solved.

## Current Limitations

- Stripe webhooks do not automatically recover full browser session identity
- cross-channel attribution remains basic
- not every CTA surface uses the new tracked URL builders yet
- server-confirmed events intentionally cover only a narrow, high-confidence set of milestones
